\subsection{Verus}

Verus is a SMT-based verification tool for \textit{Rust-like} code \cite{lattuada_verus_2023-1,verus-lang_goals_2023}, whose development has started in April of 2021 \footnote{According to the tool's GitHub log \cite{verus-lang_add_2021}}. The development has mainly been performed at the ETH Zurich and Carnegie Mellon University, USA \cite{lattuada_verus_2023-1}.

The tool features fully automatic verification and translates the to be verified code into instructions for the underlying SMT solver Z3. The tool has been inspired by projects like Dafny, Boogie, Prusti (see Chapter \ref{cap:prusti}), Coq and more \cite{verus-lang_goals_2023}. There were several goals for designing Verus:

\begin{itemize}
    \item Provide a mathematical language for designing specifications and proofs \cite{lattuada_verus_2023-1}
    \item Provide a imperative language for designing executable code \cite{lattuada_verus_2023-1}
    \item Generate small and simple verification conditions for the SMT solver \cite{lattuada_verus_2023-1}
\end{itemize}

Note that the syntax for Verus specific annotations mentioned in the complementary paper \cite{lattuada_verus_2023-1} is outdated, as it has been overhauled in the meantime \cite{chris_hawblitzel_doc_2023}. 

\subsubsection{Usability}

Like Prusti and Creusot (see Chapter \ref{cap:prusti_usability} and \ref{cap:creusot_usability}, respectively), Verus integrates itself into the Rust compiler. In a previous version of Verus, the tool has made use of a forked version of the compiler \cite{lattuada_verus_2023-1}. This has been changed recently, and Verus now integrates itself into the standard Rust compiler \cite{andrea_lattuada_port_2023}.

The verifying process in Verus is fully automatic. In contrast to Prusti and Creusot, Verus utilizes the, from the Rust-like code generated \acs{hir}, as its foundation for reasoning \cite{chris_hawblitzel_verussourcecodemd_2021}. Verus converts the \acs{hir} into instructions for the underlying SMT solver, which then executes those and returns the results back to the user.

The source code Verus processes is split into three modes: \textit{specification} mode, in which specifications functions are declared. Those are not borrow checked and represent functional designed ghost code \cite{lattuada_verus_2023-1}. The second mode is the \textit{proof} mode, in which proof functions are stated, for example user-specific lemmas \cite{lattuada_verus_2023-1}. Those functions are borrow checked and also ghost code; they can additionally be equipped with pre- and postconditions \cite{lattuada_verus_2023-1}. The third mode is the \textit{exec} mode, which contains functions that are compiled to machine code by the Rust compiler, hence including the guarantees standard Rust promises \cite{lattuada_verus_2023-1}. In this mode, functions implemented in the specification and proof mode can be called \cite{lattuada_verus_2023-1}. Furthermore, those functions can also be equipped with pre- and postconditions \cite{lattuada_verus_2023-1}, as being illustrated in \autoref{lst:verus_pre_postcondition}.

\begin{lstlisting}[float, language=Rust, caption=Example Rust code of an increase function with a pre- and postcondition for Verus, label={lst:verus_pre_postcondition}]
    exec fn incr(val: &mut u32)
        requires
            *old(val) <= u32::MAX - 1
        ensures
            *val == *old(val) + 1
    {
        *val = *val + 1;
    }
\end{lstlisting}

Source code meant for Verus has to be encapsulated inside a \lstinline{verus!} macro \cite{verus-lang_getting_2023}. Because this macro is foreign to most IDEs, syntax highlighting and suggestions are not functional. To resolve this limitation, Verus syntax support inside the macro can be added \cite{verus-lang_getting_2023} to \textit{rust-analyzer}, a language server protocol implementation for Rust, which can be used in various IDEs and code editors like Visual Studio Code or Emacs \cite{rust_analyzer_user_2023}.

\subsubsection{Technology}
Verus translates the Rust-like code into instructions for the Z3 SMT solver \cite{lattuada_verus_2023-1}. Functions written in specification mode are directly translated to SMT function calls, as Verus aims to provide lightweight verification condition encoding \cite{lattuada_verus_2023-1}. As a result, specification functions cannot be equipped with pre- and postconditions \cite{lattuada_verus_2023-1}. To support the user while encountering a verification error, Verus allows to equip those functions with \textit{soft} preconditions, annotated with \lstinline{recommends}, which Verus only invokes if there is a such an error \cite{lattuada_verus_2023-1}. 

Verus relies on Rust's borrow checker \cite{lattuada_verus_2023-1}, analog to Creusot (see Chapter \ref{cap:creusot_technology}). The tool does not create additional theories for Z3 to proof the guarantees standard Rust promises. This is apparent in the process how Verus encodes mutable references: those are encoded to Z3 as immutable values, one assignment per mutation \cite{lattuada_verus_2023-1}, therefore fully trusting the Rust compiler for correct memory safety. 

\subsubsection{Feedback}
Due to Verus' integration into the Rust compiler, error messages regarding the tools syntax or verification errors are presented like regular Rust compiler errors, similar to Prusti's behaviour (see Chapter \ref{cap:prusti_feedback}). The verification error messages are vague, as the user is not directly informed on what exactly is causing the verification to fail. As can be seen for example in \autoref{lst:verus_wrong_postcondition}, Verus marks the postcondition which is not satisfied, but is not capable to exactly pinpoint the operation in the function body, which causes the error.

\begin{lstlisting}[float, language=Rust, caption=Example output of a verification error in Verus, label={lst:verus_wrong_postcondition}]
error: postcondition not satisfied
  --> src/main.rs:9:5
   |
8  |               *val == *old(val) + 2
   |               - failed this postcondition
9  | /     {
10 | |         *val = *val + 1;
11 | |     }
   | |_____^ at the end of the function body

error: aborting due to previous error

verification results:: verified: 1 errors: 1
\end{lstlisting}

\subsubsection{Language Support and Verification Conditions}

Verus supports recursive function calls in all of its three code modes \cite{lattuada_verus_2023-1}. The user has to ensure that recursive function calls in specification and proof mode terminate, as they are only valid then \cite{lattuada_verus_2023-1}. Verus enforces this with a required \lstinline{decreases} clause, which marks the variable the the next recursive call has to decrease \cite{lattuada_verus_2023-1}. Code written in exec mode is not affected by this requirement \cite{lattuada_verus_2023-1}.

Verus does not support unsafe Rust \cite{lattuada_verus_2023-1}. It aims to replace some unsafe operations with core primitives \cite{lattuada_verus_2023-1}. For example, Verus offers so called \textit{permission pointers} and \textit{permission plus data} types as an alternative for raw heap pointers in unsafe Rust. Verus proofs the correctness of those replacements with a logic called \textit{linear ghost state} \cite{lattuada_verus_2023-1}. In the complementary literature \cite{lattuada_verus_2023-1,lattuada_verus_2023}, the authors of Verus prove the capabilities of their alternative by implementing a double-linked list, which is possible, but not practicable to implement in safe Rust \cite{learning_rust_with_entirely_too_many_linked_lists_final_2023}. 

Verus supports interior mutability \cite{lattuada_verus_2023-1}. The tool provides two strategies to safely use this technique: first, it offers an alternative to the UnsafeCell type \cite{lattuada_verus_2023-1}, which safe wrappers like Rc or Cell depend on. Its behaviour is analog to the alternative raw heap pointers Verus offers \cite{lattuada_verus_2023-1}. Second, the tool provides a type called \textit{InvCell}, which does not keep track of its interior value \cite{lattuada_verus_2023-1}. It uses invariants to restrict the valid values and enforces those invariants when a value is written into the cell \cite{lattuada_verus_2023-1}. 
The authors provide for the latter a function call example, which uses memoization across multiple clients while using interior mutability for the cache \cite{lattuada_verus_2023-1,lattuada_verus_2023}.

Verus supports reasoning for concurrent data structures \cite{lattuada_verus_2023-1}. It uses \textit{user-defined ghost values}, which in summary, represent a thread-local view of a global program state, enhanced with invariants and ghost types \cite{lattuada_verus_2023-1}. The authors provide four examples for using that technique, including a concurrent FIFO queue and a thread-safe reader-writer lock \cite{lattuada_verus_2023-1,lattuada_verus_2023}.

Additionally, Verus has support for
\begin{itemize}
    \item Pre- and postconditions \cite{lattuada_verus_2023-1}
    \item Inline assertions and assumptions \cite{lattuada_verus_2023-1,verus-lang_assert_2023}
    \item Quantifiers \cite{lattuada_verus_2023-1,verus-lang_quantifiers_2023}
    \item Loop invariants \cite{lattuada_verus_2023-1,verus-lang_loops_2023}
    \item Overflow/Underflow checks \cite{verus-lang_integers_2023}
    \item Equivalence relation \cite{verus-lang_equality_2023}
\end{itemize}

\subsubsection{Development}
Verus has been developed as a research project at the ETH Zurich, Carnegie Mellon University in the USA, UNSW Sydney, VMWare Research and Microsoft Research \cite{lattuada_verus_2023-1} and was first introduced at the 2022 occurrence of the Rust Verification Workshop \cite{peter_muller_rust_2022}. The project members involved in the development were Andrea Lattuada, Travis Hance, Chanhee Cho, Matthias Brun, Isitha Subasinghe, Yi Zhou, Jon Howell, Bryan Parno and Chris Hawblitzel \cite{lattuada_verus_2023-1}. The project is in part funded by VMWare, the Alfred P. Sloan Foundation, Intel Corporation and the Future Enterprise Security initiative at Carnegie Mellon CyLab \cite{lattuada_verus_2023-1}.

\subsubsection{Activity}
Verus is being actively developed with over 1,750 commits in the main branch since 2021 \cite{verus-lang_verus-langverus_2023}. 

Verus is not feature complete yet \cite{verus-lang_verus-langverus_2023}. This lack of feature-completeness is also indicated by the several grayed-out chapters in the complementary user manual \cite{verus-lang_getting_2023}. There are currently about 40 issues open in GitHub and about 150 have been closed already \cite{verus-lang_issues_2023}.

Verus has been evaluated with 12 different example programs, which can be found, among other examples, in the supplementary material \cite{lattuada_verus_2023}. Those examples had a total amount of 834 lines of code in specification mode, 487 lines of code in proof mode and 811 lines of code in exec mode.

Brun has used Verus to verify paging functions written in Rust for the research operating system NrOS, targeting x86\_64 architecture \cite{brun_verified_2022}.

\subsubsection{Scientific activity}
Verus has been presented at the 2022 occurrence of the Rust Verification Workshop \cite{peter_muller_rust_2022}. It also has been featured by the Rust Formal Methods Interest Group \cite{rust_formal_methods_interest_group_tools_2021}.

The first scientific publication mentioning Verus \cite{lattuada_verus_2023-1} has been cited three times in total, with one citation for the extended version and two for the normal version, according to Google Scholar at the time of writing.

\subsubsection{Documentation}
Verus provides a dedicated user guide including code examples for various features of the tool \cite{verus-lang_verus_2023}. The guide is a work-in-progress, as can be seen by the several grayed-out chapters inside of it. 

The tool provides 14 example programs in its supplementary material \cite{lattuada_verus_2023}, which demonstrate the practical appliance of Verus.
